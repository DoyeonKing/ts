# 真正的 3D 选座实现方案

本文档说明如何在剧场平台中实现**真实 3D 选座**（可旋转、缩放、点击座位选座），并区分 **H5** 与 **微信小程序** 两套技术路线。

---

## 一、整体思路


| 端            | 方案                                 | 说明                                                                                      |
| ------------ | ---------------------------------- | --------------------------------------------------------------------------------------- |
| **H5 / App** | renderjs + Three.js                | 视图层直接跑 Three.js，用 WebGL 渲染 3D 剧场，轨道控制器旋转/缩放，射线检测点选座位。                                   |
| **微信小程序**    | canvas type="webgl" + three-weixin | 小程序无 DOM，需用小程序移植版 Three.js（如 three-weixin、threejs.miniprogram）在 canvas 上渲染，逻辑与 H5 场景一致。 |


两端的**场景逻辑一致**：建舞台、排座位、射线检测、选座状态；仅 **渲染环境**（DOM+WebGL vs canvas+小程序 WebGL）和 **引入 Three 的方式** 不同。

---

## 二、H5 端实现（renderjs + Three.js）

### 2.1 为何用 renderjs

- uni-app 的 **renderjs** 运行在视图层，可操作 DOM、使用 window，且 **H5 端逻辑层与视图层在同一环境**，可直接 `import 'three'` 或通过 CDN 加载 Three.js。
- 官方说明：f2、threejs 等库均可通过 renderjs 在 App/H5 使用。
- 文档：[renderjs | uni-app](https://uniapp.dcloud.net.cn/tutorial/renderjs.html)

### 2.2 场景结构（与安卓 3D 选座对应）

1. **场景 (Scene)**：根容器。
2. **相机 (PerspectiveCamera)**：透视相机，可放在剧场后方偏上，便于俯视座位区。
3. **光照**：环境光 + 平行光，让座位和舞台有明暗。
4. **舞台**：一个扁平的 Box 或 Plane，放在 y=0 前端，材质可偏深色（如棕色）。
5. **座位**：多个小 Box（或 Plane）按排/列排成弧形或矩形，每个座位一个 Mesh，并保存 `userData.seatId`（如 "A1","B2"）和 `userData.sold`。
6. **渲染器 (WebGLRenderer)**：挂到页面中的一个 div 上，`renderer.domElement` 即 canvas。

### 2.3 交互

- **旋转/缩放**：使用 `OrbitControls`（Three.js 自带），绑定到 camera 和 domElement，实现拖拽旋转、滚轮缩放。
- **选座**：监听点击事件，将点击坐标转为标准化设备坐标（-1~1），用 `Raycaster.setFromCamera()` 和 `Raycaster.intersectObjects(seatMeshes)` 得到被点击的座位，切换其材质颜色（可选/已选），并通过 `this.$ownerInstance.callMethod('onSeatClick', [seatId])` 把 seatId 回传给 Vue 层，更新 `selectedSeats`。

### 2.4 数据流

- **Vue → renderjs**：通过 `:prop="{ selectedSeats, soldSeats }"` 传入已选座位、已售座位；renderjs 内根据 `selectedSeats`/`soldSeats` 更新对应 Mesh 的材质（已选高亮、已售灰）。
- **renderjs → Vue**：点击座位时 `this.$ownerInstance.callMethod('onSeatClick', [seatId])`，Vue 的 `onSeatClick(seatId)` 里对 `selectedSeats` 做增/删。

### 2.5 本项目中 H5 的用法

- 已提供 **H5 专用 3D 选座页**：`pages/seat-selection-3d-gl/seat-selection-3d-gl.vue`（或在同一 3D 选座页用 `#ifdef H5` 嵌入 renderjs 区块）。
- H5 运行：`npm run dev:h5`，从「选座方式」选「3D 选座」进入即可看到真实 3D 剧场与选座。

## 三、微信小程序端实现（canvas + three-weixin）

### 3.1 为何不能直接用 renderjs

- renderjs **不支持微信小程序**，小程序没有 DOM/BOM，视图层与逻辑层隔离。
- 小程序提供的是 **Canvas 2D / WebGL** 接口，需在 **canvas** 上自己跑渲染循环和事件。

### 3.2 推荐库

- **three-weixin**（Gitee）：Three.js 的微信小程序/小游戏适配版。  
[https://gitee.com/threejs/three-weixin](https://gitee.com/threejs/three-weixin)
- **threejs.miniprogram**（GitHub）：社区移植版，带 raycaster 等示例。  
[https://github.com/yannliao/threejs-example-for-miniprogram](https://github.com/yannliao/threejs-example-for-miniprogram)

### 3.3 小程序中的步骤概要

1. **引入 three-weixin**
  - 将 three-weixin 的构建产物拷贝到项目中（如 `static/libs/three-weixin.js`），或在支持 npm 的小程序工程中按官方方式引入。
2. **WXML 中放 canvas**
  ```html
   <canvas type="webgl" id="gl" canvas-id="gl" style="width:{{width}}px;height:{{height}}px;" 
           bindtouchstart="onTouchStart" bindtouchmove="onTouchMove" bindtouchend="onTouchEnd"></canvas>
  ```
3. **注册 canvas**
  - 在 Page 的 `onReady` 里用 `wx.createSelectorQuery()` 拿到 canvas 节点。
  - 调用 three-weixin 的 `THREE.global.registerCanvas(canvas)`，以便 Three.js 使用该 canvas 的 WebGL 上下文。
4. **创建场景（与 H5 一致）**
  - Scene、PerspectiveCamera、WebGLRenderer（使用 three-weixin 提供的 renderer 或适配器）。
  - 舞台 + 多排座位 Mesh，每个座位 `userData.seatId` / `userData.sold`。
  - 光照、材质与 H5 端保持一致，便于复用“场景结构”的思维。
5. **渲染循环**
  - 使用 `canvas.requestAnimationFrame`（或 three-weixin 封装）在每帧执行 `renderer.render(scene, camera)`。
6. **触摸与射线检测**
  - 在 `onTouchStart` / `onTouchEnd` 里拿到触摸点，转换为标准化设备坐标，用 `Raycaster.setFromCamera()` + `intersectObjects(seatMeshes)` 得到被点中的座位，切换状态并调用逻辑层方法更新 `selectedSeats`（如 `this.$ownerInstance.callMethod('onSeatClick', [seatId])` 或通过 setData 与逻辑层通信）。
7. **旋转/缩放**
  - 可用单指拖拽更新 camera 的 orbit 角度，双指缩放更新 camera 的 distance 或 fov，逻辑与 OrbitControls 类似，需自己写或使用 three-weixin 若提供的控制封装。
8. **内存与清理**
  - 在 `onUnload` 里取消 requestAnimationFrame、调用 `THREE.global.unregisterCanvas(canvas)`（或库提供的清理 API），并 dispose 几何体/材质/纹理，避免内存泄漏。

### 3.4 与 H5 的复用

- **场景结构、座位布局、座位 ID 规则、选座/已售状态** 可抽象成同一份“配置”（如几排几列、哪些 seatId 已售），H5 与小程序共用。
- **仅渲染与事件** 不同：H5 用 renderjs + DOM + 标准 Three；小程序用 canvas + three-weixin + 触摸转射线。

## 四、场景与座位布局建议（两端通用）

- **舞台**：宽约 20 单位，深约 2 单位，放在 z 负半轴前端。
- **座位**：多排（如 A～J），每排 15～20 个，按弧形或梯形排列；每个座位一个 Box(0.8, 0.6, 0.5) 或类似尺寸，间距约 1。
- **座位 ID**：`A1`～`A20`, `B1`～`B20`, …，与 2D 选座、订单一致。
- **材质**：可选（绿/灰）、已选（蓝/紫）、已售（红/暗灰），用 `MeshStandardMaterial` 或 `MeshBasicMaterial` 即可。

## 五、后续可优化

- 座位从**后端接口**拉取（可售/已售），前端只负责渲染与点击上报。
- 剧场模型用 **GLB/GLTF** 替换简单 Box，需在 Three 中加载模型并挂到 Scene。
- 小程序端若性能吃紧，可减少同屏座位数量或使用 LOD。

按上述方案，**H5 已具备真实 3D 选座**；**小程序端**在接入 three-weixin 并完成 canvas 注册与触摸射线后，即可复用同一套“剧场 + 选座”逻辑，实现真正的 3D 选座。

## 六、技术选型过程与踩坑记录

这个模块前期**真正花时间的不是页面编写，而是技术选型**。因为我们做的不是普通 2D 座位图，而是要在 uni-app 场景下实现可旋转、可点击、可切换第一视角的 3D 剧场，这意味着必须同时考虑：

- 跨端框架对 3D 的支持程度
- 小程序端的 WebGL / canvas 能力
- Three.js 在不同小程序生态下的适配方案
- npm 构建、运行目录、调试链路是否稳定

### 6.1 第一轮：先调研过字节系自研 3D 渲染技术路线（Oasis Engine）

项目本身预留了 `mp-toutiao` 构建脚本，说明一开始就考虑过**字节生态方向的兼容可能性**。不过我们前期关注的重点，并不只是“抖音小程序”本身，而是**字节系自研的 3D 渲染技术路线 `Oasis Engine`**，希望直接借助较成熟的实时渲染能力，更快完成沉浸式剧场场景的搭建。

这个阶段我们主要关注的是：

- `Oasis Engine` 这类引擎能力能否承接剧场空间、座位模型和灯光效果
- 现成引擎能力能否减少底层场景初始化和交互开发成本
- 与 uni-app 当前工程结构、小程序运行环境之间是否容易对接
- 调试链路、构建方式和后续维护成本是否可控

但实际推进时，遇到的核心问题是：

- 环境配置链路比较绕，跑通成本高
- 与现有工程结构结合时，适配成本偏高
- 相关资料、示例和可直接迁移的经验并不充分
- 对我们当前“先稳定落地微信小程序演示”的目标来说，投入产出比不高

所以这条路线虽然前期花了不少时间调研和尝试，但最终没有作为主实现方案。

### 6.2 第二轮：还尝试过 LayaAir 等 3D 引擎方案

在放弃 `Oasis Engine` 之后，我们还尝试过 **LayaAir** 这类更偏引擎化的 3D 方案，想看看能不能直接利用现成的 3D 场景能力来完成剧场、座位和视角切换。

这类方案的优点是：

- 引擎能力比较完整
- 对 3D 场景、材质、相机控制有较成熟的封装
- 理论上适合做更重的沉浸式空间表现

但落到我们这个项目里，问题依然比较明显：

- 和当前 uni-app 工程体系结合不够顺畅
- 对微信小程序端的适配与调试成本依然较高
- 我们实际需要的是“稳定可交互的选座场景”，而不是完整游戏引擎能力
- 对课程项目来说，学习和接入成本偏大，开发效率不理想

因此，LayaAir 这条路线也做过尝试，但最终同样没有跑成可稳定复用的主方案。

### 6.3 第三轮：尝试微信小程序侧的 Three.js 适配方案

在确定主要演示端还是**微信小程序**之后，我们又把注意力放回微信生态，对比了几种更贴近 Three.js 的方案：

1. **renderjs + Three.js**
  - 这条路线更适合 H5 / App。
  - 优点是标准 Web 环境下接入直观，Three.js 生态完整。
  - 但 `renderjs` 本身**不支持微信小程序**，所以它不能直接解决我们的小程序 3D 落地问题。
2. **three-weixin**
  - 这是常见的微信小程序 Three.js 适配方案之一。
  - 优点是思路清晰，和标准 Three.js 的场景结构比较接近。
  - 但在实际工程里，接入、注册 canvas、构建与兼容细节仍然比较繁琐。
3. **threejs-miniprogram**
  - 这是最后真正落地到项目里的方案。
  - 它更贴近微信小程序的运行方式，能在 `canvas type="webgl"` 上完成场景、相机、灯光、射线选座等核心能力。

### 6.4 最终为什么选 `threejs-miniprogram`

最后我们没有继续追“更新的技术”，而是优先选择**能稳定跑通并完成演示**的方案。综合比较后，`threejs-miniprogram` 更适合当前项目，原因主要有三点：

1. **更贴合微信小程序主战场**
  我们最终展示和体验的重点就在微信小程序端，因此优先保证微信侧稳定。
2. **Three.js 思路保留较完整**
  依然可以按 Scene、Camera、Renderer、Mesh、Raycaster 这套标准 3D 思维来搭建剧场和选座逻辑。
3. **虽然配置麻烦，但跑通后可控性更高**
  一旦 npm 构建、canvas 节点获取、渲染初始化这些问题解决后，后续的座位布局、第一视角、遮挡模拟都可以自己掌控。

### 6.5 这一轮选型给我们的经验

这部分也可以作为项目复盘总结：

- 3D 模块最难的地方，往往不是“怎么画一个座位”，而是“在哪个平台上能稳定跑起来”
- 做跨端 3D 时，**平台能力差异**比普通页面开发更明显
- 最终方案不一定是“最新方案”，而应该是**最适合当前工程、最能稳定落地的方案**

因此，这个模块的开发过程可以概括为：**先调研字节系自研 3D 渲染技术 `Oasis Engine`，再尝试 `LayaAir` 这类引擎化方案，最后比较微信生态中的 three-weixin 与 threejs-miniprogram，并选择 `threejs-miniprogram` 完成实际落地。**

## 七、本项目小程序端已实现（threejs-miniprogram）

当前工程已在**微信小程序**内实现真实 3D 选座，使用 npm 包 **threejs-miniprogram**（微信官方适配的 Three.js 0.108）。

### 6.1 使用步骤

1. **安装依赖**（项目根目录 `uni_app` 下执行）
  ```bash
   npm install
  ```
2. **微信开发者工具中构建 npm**
  - 用微信开发者工具打开 `uni_app` 编译产出目录（如 `unpackage/dist/dev/mp-weixin`）。  
  - 菜单栏：**工具 → 构建 npm**，等待构建完成。
3. **运行**
  - 本地开发：在项目根目录执行 `npm run dev:mp-weixin`，用微信开发者工具打开生成的 `unpackage/dist/dev/mp-weixin`。  
  - 进入：购票 → 某场演出 → 立即购票 · 选座 → **3D 选座**，即可看到 WebGL 3D 剧场，拖拽旋转视角、点击座位选座。

### 6.2 代码位置

- **页面**：`uni_app/pages/seat-selection-3d/seat-selection-3d.vue`  
- **条件编译**：`#ifdef MP-WEIXIN` 内为小程序 WebGL 逻辑（canvas、createScopedThreejs、场景、射线选座）；`#ifndef MP-WEIXIN` 内为其他端（如 H5 外的模拟 3D 区域选座）。  
- **依赖**：`package.json` 中已加入 `threejs-miniprogram`。

### 6.3 若构建 npm 后仍报错

- 确认微信开发者工具 **详情 → 本地设置** 中已勾选 **使用 npm 模块**。  
- 确认 **基础库版本** ≥ 2.7.0（canvas.requestAnimationFrame 等需 2.7+）。  
- 若提示找不到 `threejs-miniprogram`，可在微信开发者工具中再次执行 **工具 → 构建 npm**。
