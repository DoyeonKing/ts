# 剧场复杂算法落地方案：从爬虫到推荐闭环

这份文档单独回答一个问题：

如何把“剧场垂直推荐”真正做成一套看起来像真实系统、并且工程上能落地的算法链路。

目标不是空讲架构，而是从数据获取、数据整理、知识建模、召回排序、RAG 约束、接口输出一路打通。

---

## 一、最终要做成什么

最终系统建议定义为：

**一个面向剧场场景的垂直推荐系统，基于公开页面采集 + 人工校验的事实数据、标签体系、术语体系、知识图谱和结构化票务信息，为用户输出可解释、可约束、可执行的观剧方案。**

最终结果要具备 5 个特征：

1. 用户能用自然语言提需求
2. 系统能给出 3 到 5 个候选剧目
3. 每个结果都能解释“为什么推荐”
4. 票价、场次、城市、剧场必须来自真实数据表
5. 输出风格像 AI，但事实层不能胡编

---

## 二、整体算法链路

推荐链路建议固定成下面 8 步：

1. 爬虫/采集层获取公开演出数据
2. 数据清洗与结构化入库
3. 领域标签与术语体系整理
4. 图谱关系构建
5. 用户查询 NLU 解析
6. 多路召回
7. 事实过滤与加权排序
8. RAG 上下文拼装与结构化输出

一句话概括：

`公开数据采集 -> 知识整理 -> 图谱建模 -> 意图解析 -> 召回 -> 排序 -> 事实约束生成`

---

## 三、数据获取层

### 3.1 数据来源建议

优先采集公开页面，不要先碰复杂反爬站。

建议来源：

- 剧院官网
- 剧场公众号文章页
- 官方演出介绍页
- 公开票务详情页
- 剧目介绍页

优先采集字段：

- 剧名
- 剧种
- 简介
- 演员/卡司
- 剧场
- 城市
- 开演时间
- 结束时间
- 票价区间
- 售票状态
- 海报图
- 官方文案

### 3.2 不建议一开始采什么

- 大规模用户评论
- 必须登录才能看的完整票务数据
- 高并发采集
- 动态签名接口

### 3.3 采集目标格式

每一条原始数据建议先落成 JSON：

```json
{
  "source": "official_site",
  "source_url": "https://example.com/play/123",
  "title": "雷雨",
  "city": "北京",
  "venue_name": "北京人民艺术剧院",
  "start_time": "2026-05-01 19:30:00",
  "end_time": "2026-05-01 22:00:00",
  "min_price": 180,
  "max_price": 380,
  "status": "ON_SALE",
  "description": "剧目简介……",
  "cast_text": "主演：某某某",
  "poster": "https://example.com/poster.jpg"
}
```

## 四、数据整理层

### 4.1 事实层和语义层必须拆开

#### 事实层

放真实、可核验的信息：

- 票价
- 场次
- 城市
- 场馆
- 开演时间
- 售票状态

这部分进入：

- `performance`

#### 语义层

放推荐和分析用的信息：

- 剧种标签
- 风格标签
- 表演标签
- 术语关联
- 适合人群
- 观演门槛

这部分进入：

- `knowledge_node`
- `knowledge_edge`

### 4.2 剧目卡片标准

每部剧都建议整理成一张卡片，最少包含：

- `title`
- `genre`
- `keywords`
- `style_tags`
- `performance_tags`
- `audience_tags`
- `emotion`
- `entry_level`
- `brief_reason`
- `full_reason`

---

## 五、知识建模层

### 5.1 节点设计

节点类型：

- `PLAY`
- `ACTOR`
- `TERMINOLOGY`
- `TAG`
- `VENUE`

### 5.2 边关系设计

建议保留：

- `PERFORMS_IN`
- `HAS_TAG`
- `HAS_TERMINOLOGY`
- `PERFORMED_AT`
- `SIMILAR_TO`
- `WORKS_AT`
- `RELATED_TO`

### 5.3 标签体系

至少做这几类：

- 剧种标签：话剧、音乐剧、歌剧、芭蕾、舞剧
- 表演标签：演技爆发力强、情绪张力强、台词密度高、群像戏
- 体验标签：新手友好、情侣适合、适合带长辈、烧脑、催泪
- 预算标签：300元内可选、性价比高

### 5.4 术语体系

至少做这几类：

- 表演术语：独白、旁白、对手戏、爆发戏、收放、台词节奏
- 舞台术语：走位、调度、舞美、转台、灯光切点
- 歌剧术语：咏叹调、宣叙调、重唱
- 芭蕾术语：足尖、托举、双人舞
- 观演黑话：二刷、官摄、神卡司

---

## 六、查询理解层

### 6.1 NLU 要抽什么

用户输入：

`我想周末在北京看一场演技有爆发力的话剧，预算300内，最好适合第一次看。`

要抽出：

- `city = 北京`
- `weekendOnly = true`
- `maxPrice = 300`
- `genre = 话剧`
- `softTags = [演技爆发力强, 新手友好]`
- `terms = [爆发戏]`
- `needActorFocus = true`

### 6.2 做法

两层并行：

1. 规则解析

- 预算、城市、时间、剧种

1. 语义标签映射

- “爆发力” -> `演技爆发力强`
- “张力” -> `情绪张力强`
- “台词” -> `台词密度高`
- “新手” -> `新手友好`
- “情侣” -> `情侣适合`
- “长辈” -> `适合带长辈`
- “烧脑” -> `烧脑`
- “催泪” -> `催泪`

---

## 七、召回层

### 7.1 多路召回策略

并行做 4 路：

1. 标签召回

- 用户标签命中 `TAG` 节点

1. 术语召回

- 用户术语命中 `TERMINOLOGY` 节点

1. 用户偏好召回

- `play_rating`
- `actor_like`

1. 相似剧目召回

- `SIMILAR_TO`

### 7.2 召回结果示例

比如用户要“演技爆发力强、台词密度高、预算300内的话剧”：

- `哈姆雷特`
- `李尔王`
- `雷雨`
- `茶馆`

先召回，再由事实层过滤。

---

## 八、排序层

### 8.1 排序公式

建议用可解释加权分：

```text
final_score =
graph_score
+ preference_score
+ rating_score
+ popularity_score
+ rule_score
```

### 8.2 建议初始权重

- 标签命中：`+3.5`
- 术语命中：`+2.2`
- 喜欢演员：`+3.0`
- 历史高分偏好：`+0.8 * 用户评分`
- 同城：`+1.0`
- 周末匹配：`+0.5`
- 预算内：`+1.2`
- 超预算：`-1.5`
- 特殊标签强化：
- `演技爆发力强`：`+1.2`
- `情绪张力强`：`+1.0`
- `台词密度高`：`+0.8`
- `新手友好`：`+0.8`
- `情侣适合`：`+0.6`
- `适合带长辈`：`+0.6`
- `烧脑`：`+0.7`
- `催泪`：`+0.7`

### 8.3 为什么这样设计

因为它满足三个要求：

1. 简单
2. 可解释
3. 适合答辩

---

## 九、事实约束层

### 9.1 必须硬过滤

在排序前或排序时做：

- 没有可售场次，过滤
- 超预算，过滤或大幅降权
- 城市不匹配，过滤
- 时间不匹配，过滤

### 9.2 原则

所有事实型字段必须来自 `performance`：

- `venue`
- `city`
- `showTime`
- `priceRange`
- `status`

不允许大模型自由生成这些值。

---

## 十、RAG 生成层

### 10.1 RAG 输入应包括什么

- 用户需求摘要
- 预算/城市/时间约束
- 命中的标签与术语
- 候选剧目事实表
- 候选剧目的专业分析摘要

### 10.2 提示词约束

核心约束必须写清：

```text
你只能依据提供的候选剧目事实生成推荐，不得编造不存在的票价、场次、城市、场馆和剧目信息。
```

### 10.3 输出结构

每个方案建议返回：

- `playName`
- `venue`
- `city`
- `showTime`
- `priceRange`
- `reason`
- `analysis`
- `scoreBreakdown`

---

## 十一、爬虫到算法的工程拆分

### 角色 A：采集与清洗

- 爬公开页面
- 输出 JSON
- 清洗字段
- 导入 `performance`

### 角色 B：知识整理

- 做标签体系
- 做术语体系
- 补 `knowledge_node`
- 补 `knowledge_edge`

### 角色 C：推荐算法

- 写 `parseIntent`
- 写召回逻辑
- 写排序逻辑
- 写 RAG 上下文拼装

---

## 十二、你现在项目里对应到哪

你们当前项目已经有：

- `performance` 事实表
- `knowledge_node` / `knowledge_edge`
- `SmartRecommendationService`
- `VerticalFeatureService`
- `AiPipelineService`
- `/api/ai/chat`

所以当前真正缺的不是“重新发明算法”，而是：

1. 更稳定的数据来源
2. 更完整的标签和术语
3. 更细的权重与解释
4. 更真实的采集链路

---

## 十三、最推荐的落地节奏

### 第一步：先做可跑闭环

- 跑通采集
- 跑通入库
- 跑通标签召回
- 跑通事实过滤

### 第二步：再做更像真的

- 增加更多剧目卡片
- 增加更多标签/术语
- 优化权重
- 接 LLM 做解释生成

### 第三步：最后再谈模型升级

- 本地小模型
- LoRA 微调
- 高级图谱推荐模型

---

## 十四、答辩时最稳的口径

不要说：

“我们已经完成了海量语料 LoRA 微调，并稳定优于通用模型。”

建议说：

“我们当前实现的是基于公开演出数据采集、垂直标签与术语体系、知识图谱召回、票务事实过滤和 RAG 约束生成的剧场智能推荐链路。系统优先保证推荐结果的专业性、可解释性和事实真实性；后续会在此基础上继续扩充数据规模，并推进更强的领域模型适配。”

这句话最稳。
