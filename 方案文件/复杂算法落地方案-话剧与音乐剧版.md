# 剧场复杂算法落地方案：话剧与音乐剧双主线版

这版方案专门面向你现在真正要做的范围：

- `话剧`
- `音乐剧`

目标不是泛泛做“剧场推荐”，而是做一个更聚焦、更像真实系统的：

**话剧与音乐剧智能推荐闭环**

---

## 一、项目重新定义

项目建议重新定义为：

**一个面向话剧与音乐剧场景的垂直智能推荐系统，基于公开演出数据采集、领域标签体系、术语体系、知识图谱与真实场次约束，为用户输出可解释、可执行的观剧方案。**

这个定义的好处是：

1. 聚焦，不会太散
2. 数据更容易整理
3. 标签和术语更容易做深
4. 推荐结果更像真的

---

## 二、为什么只聚焦话剧和音乐剧

因为这两类最适合你们现在的系统能力：

### 话剧

适合做：

- 台词密度
- 表演爆发力
- 情绪张力
- 群像戏
- 经典改编
- 新手友好

### 音乐剧

适合做：

- 唱段突出
- 卡司吸引力
- 舞段完成度
- 返场/SD/末场等黑话
- 情绪共鸣
- 二刷价值

这两类一起做，既有共性，也有区分度，答辩时会显得你们理解了垂直领域。

---

## 三、最终算法链路

建议固定成下面这条闭环：

1. 采集公开演出页面数据
2. 整理话剧/音乐剧剧目卡片
3. 构建标签体系与术语体系
4. 构建知识图谱
5. 用户自然语言解析
6. 标签召回 + 术语召回 + 偏好召回
7. 票务事实过滤
8. 融合排序
9. RAG 约束生成推荐结果

一句话版本：

`数据采集 -> 知识整理 -> 图谱建模 -> 意图解析 -> 多路召回 -> 事实过滤 -> 排序 -> 解释生成`

---

## 四、数据源怎么选

### 4.1 主数据源：事实层

这一层只负责真实信息：

- 剧名
- 场次
- 城市
- 场馆
- 票价
- 售票状态
- 卡司

优先来源：

- 官方演出详情页
- 剧院官网
- 票务公开页面
- 出品方公开介绍页

### 4.2 辅助数据源：语义层

这一层负责“懂戏”和“像行内话”：

- 官方宣传文案
- 豆瓣简介和题材信息
- 公开短评人工整理
- 公众号推文文案

### 4.3 原则

- `事实层` 一定要尽量真实
- `语义层` 可以人工加工
- 大模型只能润色，不负责发明事实

- `SIMILAR_TO`

## 十二、事实过滤

### 12.1 必须硬过滤

- 场次不存在，不推
- 不在售，不推
- 超预算，不推
- 城市不匹配，不推
- 时间不匹配，不推

### 12.2 原则

所有下面这些都只能来自 `performance`：

- 票价
- 时间
- 场馆
- 城市
- 状态

## 十三、RAG 输出

### 13.1 RAG 输入

- 用户原始需求

- 命中的标签
- 命中的术语
- 候选剧目事实表

### 13.2 输出内容

每个推荐方案都输出：

- 剧名
- 场馆
- 城市
- 场次
- 票价区间
- 推荐理由
- 方案分析
- 分项得分

### 13.3 核心约束句

```text
你只能依据提供的候选剧目事实生成推荐，不得编造不存在的票价、场次、城市、场馆和卡司信息。
```
