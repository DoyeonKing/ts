# 知识图谱与推荐算法说明

本文档说明本项目中**知识图谱的构建**与**基于知识图谱的推荐算法**的设计与实现。

**算法定位**：  
- **当前已实现**：基于图结构的**启发式基线算法**（一度/二度邻居 + 关系权重），无训练、纯规则，用于冷启动与兜底。  
- **目标（您已强调）**：采用**高级/论文级算法**，以 `算法/RecBole-master` 中的知识图谱推荐模型（如 KGCN、KGAT、RippleNet）为参考并接入项目。  
- **对接方式**：通过 `算法/export_for_recbole.py` 将 MySQL 中的评分与图谱导出为 RecBole 格式，在 RecBole 中训练后，将结果接入 Java 推荐接口（详见本文档**第六节**与 `算法/README_RecBole对接.md`）。

### RecBole 能否用于本项目？（结论：能）

| 维度 | RecBole 要求 | 本项目情况 | 是否匹配 |
|------|--------------|------------|----------|
| **用户–物品交互** | 需 user–item 交互表（可带评分） | 有 `play_rating`（user_id, play_id, rating） | ✅ 有，可导出为 .inter |
| **知识图谱** | 需三元组 head–relation–tail | 有 `knowledge_edge`（source_node_id, relation_type, target_node_id） | ✅ 有，可映射为 .kg |
| **物品–实体链接** | 需 item 与 KG 中 entity 的对应 | 剧目即图谱中 PLAY 节点，id 一致即可作为 link | ✅ 有，可导出为 .link |
| **技术栈** | Python / PyTorch | 后端为 Java | ⚠️ 不嵌在同一进程；用 RecBole **离线训练**，把**结果**给 Java 用 |

**结论**：RecBole **可以用于本项目**。数据上三项都满足；技术栈上 RecBole 与 Java 并存的方式是：用 RecBole 做训练/预测，将得到的推荐列表或预计算表交给现有 Java 接口使用（详见第六节）。

## 一、知识图谱数据模型

### 1.1 存储方式

- **数据库**：MySQL（关系型），未使用图数据库（如 Neo4j）。
- **表结构**：
  - **knowledge_node**：图谱中的每个实体（节点），含 `id, name, node_type, description, extra_data, created_at`。
  - **knowledge_edge**：实体之间的关系（边），含 `id, source_node_id, target_node_id, relation_type, label, weight, created_at`。

### 1.2 节点类型（node_type）

| 类型 | 说明 | 示例 |
|------|------|------|
| PLAY | 剧目 | 哈姆雷特、茶馆、天鹅湖 |
| ACTOR | 演员 | 濮存昕、袁泉、何冰 |
| TERMINOLOGY | 术语/行话 | 独白、谢幕、SD、返场、卡司 |
| TAG | 标签 | 莎士比亚、悲剧、经典、话剧 |
| VENUE | 场馆 | 国家大剧院、北京人艺 |

### 1.3 关系类型（relation_type）

| 关系 | 含义 | 示例 |
|------|------|------|
| PERFORMS_IN | 演员参演剧目 | 濮存昕 — 饰哈姆雷特 — 哈姆雷特 |
| HAS_TAG | 剧目拥有标签 | 哈姆雷特 — HAS_TAG — 莎士比亚 |
| HAS_TERMINOLOGY | 剧目涉及术语 | 哈姆雷特 — HAS_TERMINOLOGY — 独白 |
| SIMILAR_TO | 剧目相似 | 哈姆雷特 — SIMILAR_TO — 李尔王 |
| PERFORMED_AT | 剧目在某场馆演出 | 哈姆雷特 — PERFORMED_AT — 国家大剧院 |
| WORKS_AT | 演员在某场馆工作 | 濮存昕 — WORKS_AT — 北京人艺 |
| RELATED_TO | 术语等之间的相关 | 独白 — RELATED_TO — 旁白 |

种子数据通过接口 `POST /api/knowledge-graph/init` 一次性写入；后续可从业务库同步或人工维护。

## 二、推荐算法设计（当前基线：图启发式）

### 2.1 任务定义

- **输入**：当前用户正在查看的某个节点（如某部剧目、某位演员、某条术语）。
- **输出**：与该节点**相关**的推荐列表，按类型分组（如「相似剧目」「相关演员」），并附带**推荐理由**（如「同标签」「参演该剧」），便于可解释推荐。

### 2.2 算法思路概述（基线）

采用**基于图结构的启发式推荐**，不训练神经网络，完全依赖图谱拓扑与关系类型：

1. **候选集合**：从当前节点出发，取**一度邻居**（直接相连的节点）与**二度邻居**（邻居的邻居，且排除当前节点与一度邻居），避免重复与自引用。
2. **打分**：对每个候选节点，根据「通过哪类关系被触达」赋予权重；同一候选若被多条边触达，则分数累加。
3. **排序与截断**：按总分降序排序，再按节点类型分组，每种类型取 Top-N（如剧目最多 6 条、演员最多 4 条），并附带推荐理由。

### 2.3 关系类型权重

不同关系类型对「相关度」的贡献不同，用整数权重表示，权重越大推荐优先级越高：

| 关系类型 | 权重 | 说明 |
|----------|------|------|
| SIMILAR_TO | 10 | 相似剧目，强相关 |
| PERFORMS_IN | 8 | 参演关系，强相关 |
| HAS_TAG | 6 | 同标签 |
| HAS_TERMINOLOGY | 5 | 相关术语 |
| PERFORMED_AT | 5 | 同场馆演出 |
| WORKS_AT | 4 | 同场馆 |
| RELATED_TO | 3 | 一般相关 |
| 其他 | 1 | 默认 |

- **一度邻居**：按上表权重直接计入该候选节点的得分。
- **二度邻居**：按上表权重 **÷2** 再计入（间接相关，降权）。

若一条边上带有 `label`（如「饰 哈姆雷特」），优先用 `label` 作为推荐理由；否则用关系类型对应的默认文案（如「参演该剧」「同标签」）。

### 2.4 流程小结（伪代码）

```
输入：当前节点 nodeId，允许推荐的节点类型 allowTypes，各类型数量上限
1. 查询所有与 nodeId 相连的边 edges
2. 对每条边，得到邻居 otherId（≠ nodeId），按 relation_type 取权重 w，累加到 scores[otherId]，并记录理由
3. 对当前所有已得分的节点（一度邻居）mid，再查与 mid 相连的边，得到二度邻居 otherId
   - 若 otherId 已是 nodeId 或一度邻居，则跳过
   - 否则按 relation_type 权重/2 累加到 scores[otherId]，理由为「间接相关」
4. 将 scores 按分数降序排序，得到 sortedIds
5. 按 sortedIds 顺序，根据节点类型将节点填入 plays、actors、others 等列表，直至达到各类型数量上限
6. 返回 { plays, actors, others, source }
```

## 三、接口与前端使用

### 3.1 推荐相关接口

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/recommendations/play/{playId} | 根据剧目推荐：相似剧目、相关演员等 |
| GET | /api/recommendations/actor/{actorId} | 根据演员推荐：其剧目、合作演员等 |
| GET | /api/recommendations/node/{nodeId} | 根据任意图谱节点推荐（通用） |
| GET | /api/recommendations/lookup?name=xx&type=PLAY | 按名称查图谱节点 ID |

返回结构统一包含：`plays`、`actors`、`others`（列表，每项含 `id, name, nodeType, description, reason`）以及 `source`（当前节点信息）。

### 3.2 知识图谱可视化与数据

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/knowledge-graph/full | 全图节点与边，用于前端力导向图 |
| GET | /api/knowledge-graph/node/{id}/neighborhood | 某节点的一度邻居子图 |
| GET | /api/knowledge-graph/nodes?type=TERMINOLOGY | 按类型取节点列表（如行话科普） |
| POST | /api/knowledge-graph/init | 初始化种子数据 |

### 3.3 前端展示位置

- **剧目详情页**：根据剧目 ID 调「根据剧目推荐」，展示「猜你喜欢」中的相似剧目、相关演员。
- **演员详情页**：根据演员 ID 调「根据演员推荐」，展示推荐剧目、合作演员。
- **图谱可视化页**：点击某个节点后，根据节点 ID 调「根据节点推荐」，在弹窗中展示「猜你喜欢」列表，并可进入对应详情页。

前端仅需调用上述 REST 接口并渲染返回的 `plays`、`actors`、`others` 及 `reason`，无需实现任何图算法逻辑。

## 六、高级算法：RecBole 知识图谱推荐模型（对接方案）

为满足「采用高级/论文级算法、以 RecBole 开源代码为参考」的需求，已在本仓库中提供以下对接能力：

1. **数据导出脚本**  
   - 路径：`算法/export_for_recbole.py`  
   - 作用：从 MySQL 读取 `play_rating`、`knowledge_node`、`knowledge_edge`，生成 RecBole 所需的 **.inter**（用户–剧目评分）、**.kg**（图谱三元组）、**.link**（剧目–实体映射）及数据集配置。  
   - 使用：配置数据库连接后执行 `python export_for_recbole.py`，输出在 `算法/theater_recbole/`。

2. **RecBole 训练**  
   - 将 `theater_recbole` 中文件复制到 RecBole 的 `dataset/theater/`，在 `算法/RecBole-master` 下执行例如：  
     `python run_recbole.py --model=KGAT --dataset=theater`  
   - 可选模型：KGCN、KGAT、RippleNet、KGIN、MKR 等（见 RecBole 文档及 `knowledge_aware_recommender` 相关配置）。

3. **结果接回 Java**  
   - **方式 A**：离线用 RecBole 生成每个用户的 Top-N 推荐表，导入 MySQL，Java 推荐接口优先查该表（已登录用户），未命中时回退到第二节的图启发式。  
   - **方式 B**：将 RecBole 预测封装为 HTTP 服务，Java 在需要个性化推荐时调用该服务，将返回的剧目 id 列表作为推荐结果。

详细步骤、环境与配置说明见：**`算法/README_RecBole对接.md`**。  
当前 Java 端 `RecommendationService` 仍为**基线图启发式**；接入 RecBole 输出后，可在同一套推荐接口下切换或融合「高级模型结果」与「基线兜底」。
