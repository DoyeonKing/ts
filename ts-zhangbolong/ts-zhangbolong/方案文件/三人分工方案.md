# 餐剧盒剧场平台 - 三人分工方案

> 基于当前项目现状制定，明确**不做支付**、**只提供3D选座视图**，聚焦：数据库设计、知识图谱（真数据）、推荐系统、AI智能助手、文创设计、3D视图优化。

---

## 一、项目现状与约束

| 模块 | 现状 | 目标 |
|------|------|------|
| **数据库** | 未连接，数据全为假 | 完整设计并落地，统一数据源 |
| **知识图谱** | 前端有 mock 数据 | Neo4j 真实数据 + 图谱 API |
| **推荐系统** | 未做好 | 与 AI 助手结合，实现多方案推荐 |
| **3D 选座** | 前端有实现 | 仅提供 3D 视图，不做支付 |
| **支付** | ❌ 不做 | 移除或弱化 |
| **AI 智能助手** | 未实现 | 自然语言对话、多方案推荐、一条龙服务 |
| **文创设计** | 只有 UI 壳 | AI 生成海报/周边/纪念品等 |
| **盘票** | goDetail 空、publish 仅 toast | 求票/转票列表、发布、详情页、对接后端 |
| **社群** | post-detail 页面不存在 | 补全帖子详情页、对接数据 |
| **搜索** | doSearch 仅 toast，无 API | 实现搜索 API + 前端对接 |
| **我的** | onMenuClick 空、log-detail 不存在 | 功能菜单可点击、剧场日志详情页 |
| **知识图谱子模块** | 新闻资讯/时间线/智能问答 仅「开发中」 | 实现或做占位页 |
| **AI 生成（文字类）** | 观后感/剧情分析/演员评价 仅模拟 | 接入大模型 API，真实生成 |

## 二、三人分工总览

| 角色 | 主要负责 | 核心产出 |
|------|----------|----------|
| **成员A：3D + 知识图谱** | 3D 选座、知识图谱（Neo4j + 前端）、新闻/时间线/智能问答 | 3D 视图、知识图谱 API、图谱子模块 |
| **成员B：AI + 推荐** | AI 智能助手、推荐算法、文创生成、AI 文字生成（观后感等） | 对话推荐、多方案、文创图、观后感/剧情分析/演员评价 |
| **成员C：数据库 + 前端** | 数据库、基础 API、盘票、社群、搜索、我的、缺失页面 | 数据库、盘票/社群 API、post-detail/log-detail、搜索、我的菜单 |

> **说明**：成员A 负责 3D 和知识图谱，因这两块目前由同一人开发，便于延续性。

## 三、成员A：3D 选座 + 知识图谱

### 3.1 3D 选座视图

**工作内容：**
1. **校验与优化** 现有 3D 选座页面（`seat-selection-3d.vue`）
2. **对接座位 API**：调用成员C提供的 `/api/seats`，获取座位布局、可用/已售状态（仅展示，不做支付）
3. **明确只做视图**：选座后不跳支付，可展示「已选座位」+ 提示「请至线下/官方渠道购票」
4. **移除或弱化支付流程**：选座确认后不进入 payment 页，或改为「预约/咨询」类引导

**参考文档：**
- `3D_Seating_API_Design.md`
- `seat-selection-3d.vue`、`seat-mode.vue`

### 3.2 知识图谱（Neo4j）

**工作内容：**
1. **安装配置 Neo4j**
2. **设计节点与关系**：剧目、演员、术语、标签、场馆 及 参演、关联、标签等关系
3. **实现知识图谱 API**（对接 uni_app 已有调用）：
   - `GET /api/knowledge-graph/full` 全图
   - `GET /api/knowledge-graph/node/{id}/neighborhood` 邻居
   - `GET /api/knowledge-graph/search?keyword=` 搜索
   - `GET /api/knowledge-graph/nodes?type=` 按类型查询
   - `POST /api/knowledge-graph/init` 初始化种子数据
4. **数据来源**：使用成员C提供的剧目/演员/术语等基础数据，导入 Neo4j 或定时同步
5. **前端图谱页**：`graph-view.vue`、`knowledge-graph.vue`、`terminology.vue` 等，确保调用自己的知识图谱 API

**参考文档：** `知识图谱实现方案.md`、`uni_app/api/knowledgeGraph.js`

### 3.3 知识图谱子模块（新闻资讯 / 时间线 / 智能问答）

**工作内容：**
1. **新闻资讯**：实现简单资讯列表页，或对接数据源展示行业动态、演出资讯
2. **时间线**：剧目/演员发展时间线、首演与重要场次记录（可用占位页 + 简单数据）
3. **智能问答**：基于知识图谱的自然语言问答（如「哈姆雷特的主演是谁」），可调用大模型 + 图谱检索，或与成员B协作

> 若阶段紧张，可先做占位页 + 简单展示，后续迭代。

### 3.4 与成员 B、C 的对接点

- 为 **成员B** 提供：知识图谱搜索接口（RAG/检索增强时可调用）
- 调用 **成员C**：座位接口（3D 视图用）、剧目/演员等基础数据（知识图谱初始化用）

## 四、成员B：AI + 推荐（智能助手 + 文创）

### 4.1 AI 智能对话助手（核心）

**工作内容：**
1. **集成大模型 API**（豆包/文心/通义等）
2. **意图识别与实体提取**：识别「看剧」「找演员」「周边服务」等
3. **多方案生成**：
   - 根据用户输入（剧目类型、预算、时间等）生成 3–5 个观剧方案
   - 每个方案包含：剧目、场次、演员、票价区间、周边美食、交通建议
4. **RAG 检索增强**（可选）：结合知识图谱/数据库检索，减少幻觉
5. **后端对话接口**：`POST /api/ai/chat`，支持流式输出（可选）

**参考文档：**
- `AI智能推荐系统升级方案.md` 第 2.1、3.2 节
- 示例对话、多方案策略、Prompt 设计

### 4.2 推荐算法

**工作内容：**
1. **基础推荐逻辑**：基于剧目标签、演员、评分、热度
2. **与 AI 结合**：AI 生成方案时调用推荐服务，保证推荐数据来自真实库
3. **用户画像**（可选）：收藏、浏览记录 → 偏好标签 → 个性化排序

### 4.3 文创设计生成

**工作内容：**
1. **接入图像生成 API**（豆包/文心等支持文生图的模型）
2. **文创类型**：海报、周边（T恤/帆布袋等）、纪念品、明信片、手账素材
3. **Prompt 工程**：根据剧目、演员、风格等构建生成指令
4. **API**：`POST /api/ai/creative`，入参：类型、剧目、风格；出参：图片 URL 或 base64
5. **与 uni_app 的 ai-generation 页面对接**

### 4.4 AI 生成（文字类：观后感 / 剧情分析 / 演员评价）

**工作内容：**
1. **观后感**：根据剧目、用户输入，调用大模型生成观剧感受
2. **剧情分析**：生成剧目剧情结构、主题与人物分析
3. **演员评价**：生成对演员表演的点评与印象
4. **API**：`POST /api/ai/generate-text`，入参：`{ type: 'review'|'plot'|'actor', playId?, userInput? }`，出参：生成文本
5. 与 `ai-generation.vue` 中 review、plot、actor 类型对接，替换现有模拟逻辑

**参考文档：** `AI文创生成技术方案-口头回答版.md`、`ai-generation.vue` 中的 types

### 4.5 与成员 A、C 的对接点

- 调用 **成员A** 提供的知识图谱接口做检索（RAG 时可选用）
- 调用 **成员C** 提供的剧目/演员/场馆等基础接口
- 为 **成员C** 提供：对话接口、文创生成接口，供前端调用

## 五、成员C：数据库 + 前端（基础数据与页面对接）

### 5.1 数据库设计与落地

**工作内容：**
1. **设计核心表结构**（不做支付，可简化订单相关）：
   - 剧目 `plays`、演出 `performances`、场次 `schedules`
   - 演员 `actors`、剧场/场馆 `venues`
   - 座位（用于 3D 视图展示）`seats`、`sections`
   - 术语/标签 `terminologies`、`tags`
   - 盘票：`ticket_requests`（求票）、`ticket_transfers`（转票）
   - 社群：`posts`、`post_comments`（可选）
   - 剧场日志：`viewing_logs`（可选）
   - 用户画像相关（推荐用）：`user_preferences`、`viewing_history`（可选）
2. **搭建 MySQL/PostgreSQL**，按《数据库设计文档》建表
3. **准备种子数据**：剧目、演员、术语等基础数据，供知识图谱和推荐使用
4. **Spring Boot 接入数据库**，实现基础 CRUD 接口

**参考文档：**
- 项目根目录 `数据库设计文档.md`
- `未来方向指南.md` 中的建表与 Entity 示例

### 5.2 后端基础 API

**工作内容：**
1. **剧目、演员、演出、场馆** 等基础查询接口
2. **座位 API**：`GET /api/seats`（performanceId, section → 座位状态），供成员A的 3D 视图使用
3. **演出列表**：`GET /api/performances`
4. **搜索 API**：`GET /api/search`（剧目、演员、剧场关键词搜索）
5. **盘票 API**：求票/转票列表、发布求票、发布转票、详情接口
6. **社群 API**（可选）：帖子列表、帖子详情
7. 为 **成员A** 提供数据源，供知识图谱初始化/同步

### 5.3 盘票功能（求票 / 转票）

**工作内容：**
1. **ticket-exchange.vue** 对接后端：求票列表、转票列表、我的发布
2. **发布求票 / 发布转票**：实现发布表单页或弹窗，调用 `POST /api/tickets/request`、`POST /api/tickets/transfer`
3. **详情页**：补全 `goDetail(t)`，新建 `ticket-detail` 页面（或复用现有结构），展示求票/转票详情

### 5.4 社群功能

**工作内容：**
1. **补全 post-detail 页面**：新建 `pages/post-detail/post-detail.vue`，展示帖子详情（内容、评论、点赞等）
2. 在 `pages.json` 中注册该页面
3. `community.vue` 的 `goPostDetail` 已跳转，确保能正常打开详情页

### 5.5 搜索功能

**工作内容：**
1. **search.vue**：实现 `doSearch`，调用 `/api/search`，展示搜索结果（剧目、演员、剧场等）
2. 替换现有「仅 toast」逻辑

### 5.6 我的（个人中心）

**工作内容：**
1. **onMenuClick**：我的订单、收藏、历史、积分、客服、反馈 等菜单可点击，跳转对应页或占位页（不做支付则订单可为空/占位）
2. **log-detail 页面**：新建 `pages/log-detail/log-detail.vue`，展示剧场日志详情
3. **profile.vue** 的 goLogDetail、goAIDetail、goTicketDetail 等跳转需有效

### 5.7 前端页面对接

**工作内容：**
1. **AI 智能助手入口**：新增对话页或改造现有入口，调用成员B的 `/api/ai/chat`
2. **文创生成页**：`ai-generation.vue` 对接 `/api/ai/creative`，展示生成结果
3. **知识图谱页**：确保 `knowledgeGraph.js`、`graph-view.vue` 等调用成员A的接口（成员A 负责图谱 API，前端由 C 对接）
4. **首页/推荐**：对接成员B的推荐接口，展示个性化推荐内容

### 5.8 其他前端事项

**工作内容：**
1. **移除/调整支付相关**：tabBar、订单、支付页等
2. **统一请求 baseURL**：确保 `config/index.js` 指向后端地址

### 5.9 与成员 A、B 的对接点

- 为 **成员A** 提供：座位接口、剧目/演员/术语等基础数据（知识图谱初始化）
- 调用 **成员A**：知识图谱接口（前端图谱页）
- 调用 **成员B**：AI 对话、文创生成、推荐接口

## 七、接口约定（前后端对齐）

### 7.1 知识图谱（成员A 提供）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/knowledge-graph/full` | GET | 全图 nodes + edges |
| `/api/knowledge-graph/node/{id}/neighborhood` | GET | 指定节点邻居 |
| `/api/knowledge-graph/search` | GET | keyword 搜索 |
| `/api/knowledge-graph/nodes` | GET | type=PLAY/ACTOR/TERMINOLOGY 等 |
| `/api/knowledge-graph/init` | POST | 初始化种子数据 |

### 7.2 AI 服务（成员B 提供）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/ai/chat` | POST | 对话，body: `{ query, context? }` |
| `/api/ai/creative` | POST | 文创图片生成，body: `{ type, playId?, style? }` |
| `/api/ai/generate-text` | POST | 观后感/剧情分析/演员评价，body: `{ type, playId?, userInput? }` |
| `/api/ai/recommend` | POST | 推荐方案，body: `{ query, userId? }` |

### 7.3 座位/演出/基础数据（成员C 提供）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/performances` | GET | 演出列表 |
| `/api/seats` | GET | performanceId, section → 座位状态（仅展示用，供成员A 3D 视图） |
| `/api/plays`、`/api/actors`、`/api/venues` 等 | GET | 剧目、演员、场馆等基础 CRUD |
| `/api/search` | GET | 关键词搜索 |

### 7.4 盘票/社群（成员C 提供）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/api/tickets/requests` | GET | 求票列表 |
| `/api/tickets/transfers` | GET | 转票列表 |
| `/api/tickets/request` | POST | 发布求票 |
| `/api/tickets/transfer` | POST | 发布转票 |
| `/api/tickets/:id` | GET | 求票/转票详情 |
| `/api/posts/:id` | GET | 帖子详情（社群） |

## 十、协作建议

1. **每周同步**：接口变更、联调进度、阻塞点
2. **先定接口再开发**：三人一起过一遍接口字段，避免后期大改
